在一切开始之前, 先思考一个问题, 仅就”重用线程”这个首要目标来说, 该如何实现? 比较明确的几点是:

  1. 一定先创建一定数目的Thread, 负责执行我们传入的Runnable.
  2. 根据Thread本身的生命周期, 执行完就进入 Dead 状态, 所以这个Thread不能停, 需要一直处于 Running 状态, 也就是说, 需要在其中执行一个”死循环”.

除此之外, 还有一点: 如何让空闲的Thread拿到传入的Runnable? // TODO 这句没表述清楚

WorstPool和ThreadPoolExecutor的模型基本类似. 后者有这么个类成员: BlockingQueue<Runnable> workQueue;. 而Thread在其中被抽象成为类Worker.

继续深入之前, 以execute方法为切入点, 说明一下整体逻辑. 对于WorstPool来说, 逻辑很简单: 所有线程在初始化时就被创建, 队列也是固定大小, 如果放不下就抛出异常. ThreadPoolExecutor则要复杂一些, 需要提到另外类成员: corePoolSize, maximumPoolSize

A ThreadPoolExecutor will automatically adjust the pool size according to the bounds set by corePoolSize and maximumPoolSize.
When a new task is submitted in method execute, and fewer than corePoolSize threads are running, a new thread is created to handle the request, even if other worker threads are idle. If there are more than corePoolSize but less than maximumPoolSize threads running, a new thread will be created only if the queue is full.

可知, 线程(Worker)不是在类初始化时创建的, 而是在提交任务时决定是否创建的.

RunState/生命周期, shutdown, shutdownNow,awaitTermination

由上文, ctl包装的另一个值就是runState. 为了先对它有个概念, 先看下这段比较长的注释, 很痛苦:

The runState provides the main lifecyle control, taking on values: 

    - RUNNING:  Accept new tasks and process queued tasks
- SHUTDOWN: Don't accept new tasks, but process queued tasks
- STOP:     Don't accept new tasks, don't process queued tasks, and interrupt in-progress tasks
- TIDYING:  All tasks have terminated, workerCount is zero, the thread transitioning to state      TIDYING will run the terminated() hook method
- TERMINATED: terminated() has completed

下面几个方法都是对runState简单的读写操作, 几乎可以根据名称判断用处

  • runStateLessThan(int c, int s)
  • runStateAtLeast(int c, int s)
  • isTerminated()
  • isTerminating()
  • isShutdown()
  • isRunning(int c) // 是否处于RUNNING状态
  • advanceRunState(int target) // 将runState设置为目标值
    再看另一段注释:

The runState monotonically increases over time, but need not hit each state. The transitions are: 

- RUNNING -> SHUTDOWN: On invocation of shutdown(), perhaps implicitly in finalize()
- (RUNNING or SHUTDOWN) -> STOP: On invocation of shutdownNow()
* SHUTDOWN -> TIDYING:  When both queue and pool are empty
* STOP -> TIDYING:  When pool is empty
            * TIDYING -> TERMINATED: When the terminated() hook method has completed

可以看出来, 线程池初始化之后, 如果不调用shutdown, shutdownNow它是一直处于RUNNING状态的. 生命周期的变化都始于这两个方法, 来看一下这两个方法包含的大致步骤

showdown

  • 此操作调用之前提交的任务(即包含队列中的任务)都会被执行, 但是不接受新任务. 另外此方法的注释中提到: (shutdownNow也是如此)

    这句的意思是, shutdown并不会阻塞当前线程, 从而等待所有任务执行完. 如果需要的话, 使用awaitTermination. awaitTermination的实现是通过一个Condition termination实现的.的await来实现的, 逻辑比较简单, 根据runState是否达到TERMINATED状态决定是否继续await.
    有await的地方, 就有signal的地方. 在后面会提到.

这里的”ShutDown OK”马上就打印了, 但是线程池中的任务还没有完成.

shutdownNow

试图终止当前运行的任务, 并将队列中的任务全部移除. 由于试图终止的方式为interrupt, 所以实际上并不能保证一定成功.

来看一下两个方法的主要内容:

  1. advanceRunState(state)
    这里两个都有调用, 只不过目标值不同.
  2. interruptIdleWorkers vs interruptWorkers:
    两者都是试图对进行中的worker thread进行interrupt. 不同的是, 前者会先调用tryLock(). 而在runWorker的循环中, 每次执行task会先调用worker.lock, 结束后unlock. 所以说非空闲的任务不会受到影响.
  3. onShutdown
    留给子类的一个钩子, 此处基本可以忽略
  4. drainQueue
    将队列中的元素抽取到另一个List中, 并移除此元素.
  5. tryTerminate
    两者都有调用.
    此处回顾一下前面的runState转换
    • SHUTDOWN -> TIDYING: When both queue and pool are empty
    • STOP -> TIDYING: When pool is empty
      这个过程即为tryTerminate做的事情.

一般生命周期流程如下:
RUNNING (shutdown) -> SHUTDOWN -> TIDYING -> TERMINATED
RUNNING (shutdownNow) -> STOP -> TIDYING -> TERMINATED

AbortPolicy

当任务无法放到队列中, 且没超过maximumPoolSize, 那么会新开启个线程. 如果超过了maximumPoolSize, 任务会被rejected.

当Executor被shutdown后, 或者指定了有限的maximumPoolSize和queque大小, 且达到饱和状态时, 提交的任务会被rejected. 这种情况下, 会执行它的RejectedExecutionHandler中的rejectedExecution方法.

提供了四种预定义策略, 这几个非常简单, 可以自己看一下.

  1. 默认 AbortPolicy. 抛出一个runtime异常(RejectedExecutionException)
  2. CallerRunsPolicy. 将提交的runnable直接同步run, 而不是用线程池run.
  3. DiscardPolicy 非常简单, 直接放弃治疗
  4. DiscardOldestPolicy. 如果Executor没有被shutdown, 队列头的任务呗抛弃, 重试执行.
